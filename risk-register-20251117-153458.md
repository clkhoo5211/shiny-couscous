### RISK-001: URL Survey Complexity & Form Extraction Challenges
**Risk ID**: RISK-001  
**Category**: Technical / Scope  
**Probability**: High (70%)  
**Impact**: High  
**Risk Level**: üî¥ HIGH

**Description**:  
Surveying and extracting all Labuan FSA forms from 5 different URLs (PDFs and web pages) may be more complex than anticipated. Forms may have complex structures, nested fields, conditional logic, or unclear documentation.

**Impact**:
- Delayed form extraction timeline
- Incomplete form data extraction
- Incorrect form schema generation
- Rework required for form definitions

**Mitigation Strategies**:
1. **Pre-Survey Analysis**: Conduct preliminary analysis of all URLs before detailed extraction
2. **Iterative Extraction**: Extract forms in phases, starting with simplest forms
3. **Validation Checkpoints**: Review extracted data at each phase
4. **Expert Consultation**: Consult with Labuan FSA or domain experts if needed
5. **Buffer Time**: Allocate 20% buffer time for form extraction phase

**Contingency Plan**:
- If extraction takes longer than expected, prioritize most critical forms first
- Use manual data entry for complex forms if automated extraction fails
- Extend Phase 1 timeline by 1 week if necessary

### RISK-002: API-Driven Dynamic Rendering Complexity
**Risk ID**: RISK-002  
**Category**: Technical  
**Probability**: Medium-High (60%)  
**Impact**: High  
**Risk Level**: üî¥ HIGH

**Description**:  
Implementing fully dynamic form rendering where ALL DOM elements are controlled by Python backend API responses is a complex architectural pattern. This may introduce:
- Performance issues with complex forms
- Difficulties in maintaining component modularity
- Challenges in handling edge cases
- Integration complexity between frontend and backend

**Impact**:
- Extended development time
- Performance degradation
- Maintenance challenges
- Potential need for architectural changes

**Mitigation Strategies**:
1. **Reference Implementation Analysis**: Thoroughly analyze OpenMRS Form Builder and Alibaba Formily before implementation
2. **Proof of Concept**: Build POC for dynamic rendering early (Week 3-4)
3. **Component Library First**: Build base components before dynamic renderer
4. **Performance Testing**: Test with complex forms early in development
5. **Incremental Implementation**: Start with simple forms, add complexity gradually

**Contingency Plan**:
- If dynamic rendering proves too complex, implement hybrid approach (static base + dynamic fields)
- Consider using existing form library (Formily) as foundation
- Allocate additional development time if needed

**Owner**: Frontend Developer  
**Status**: ‚è≥ Active
